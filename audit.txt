1. Backend/hardware dispatch inventory
Full inventory of every device-selection site in the repo:

llm-core/src/backends/candle.rs:304-322 (CandleBackend::load_weights) — the one real dispatch point. Estimates model size on disk → HardwareProfile::get() → profile.choose_device(estimated_bytes) → matches BackendChoice::{Cuda,Metal,Cpu} → constructs the candle_core::Device. This is the sanctioned pattern, matching CLAUDE.md's rule.
llm-core/src/profile/mod.rs — HardwareProfile::detect() (lines 105-173) is the single detection point: cuda_is_available() → nvidia-smi → CUDA driver API → LLM_FORCE_CUDA opt-in → else Metal via query_metal_vram() (feature+target_os gated) → else CPU. choose_device() (180-210) applies a 15% headroom check against free VRAM/RAM and errors (not silently degrades) on CPU OOM risk. This is correctly the only place BackendChoice is decided.
All other .is_cpu()/.is_cuda()/Device::Cpu occurrences in candle.rs, vision.rs, audio.rs, weights.rs (lines 90, 122, 637 in vision.rs; 324, 329-334, 482, 908, 1242, 1845 in candle.rs; 100-101 audio.rs) operate on self.device/dev/table.device() — i.e., the already-chosen device — purely for dtype selection (F32-on-CPU vs F16-on-GPU) or CPU-side tensor round-tripping. These are bookkeeping reads, not re-dispatch. No bypass found in the real inference path.
llm-tests/*, llm-core/tests/*, llm-cli/src/bin/test_audio_encoder.rs — all Device::Cpu uses are test fixtures, irrelevant to production dispatch.
scratch/check_qmatmul_dtype.rs and scratch/test_device.rs — both contain ad hoc Device::new_cuda(0).unwrap_or(Device::Cpu) chains that bypass HardwareProfile entirely. These are throwaway debug scripts (not compiled as part of the workspace — see category 7), so not a live bug, but they are exactly the anti-pattern CLAUDE.md warns against and are committed to git, which risks someone copy-pasting the pattern into real code later.
llm-cli/src/profile.rs... no such file; llm-cli/src/bin/devices.rs only reads BackendChoice for display, doesn't decide it. Fine.
llm-cluster and llm-scheduler: no device-selection code at all (they operate on tensors/logits already produced by the chosen backend).
Verdict: The one core dispatch point (candle.rs → HardwareProfile::choose_device) is real and consistent, confirming PROGRESS.md's prior finding still holds on current master. The only bypasses are inert scratch scripts.

Critical finding — duplicated AND inconsistent chat-template logic, one of which never runs:

llm-cli/src/lib.rs:125-129 (chat_completions, the production OpenAI-compatible HTTP server — /v1/chat/completions, the actual llm serve endpoint) — hardcodes ChatML unconditionally for every model, with no architecture check, no is_gemma, no chat_template field consulted at all:


let mut prompt = String::new();
for msg in &req.messages {
    prompt.push_str(&format!("<|im_start|>{}\n{}<|im_end|>\n", msg.role, msg.content));

prompt.push_str("<|im_start|>assistant\n");
This is possible because ModelMeta is never threaded into AppState at all. In llm-cli/src/main.rs:39, backend.load_weights(path)? returns a ModelMeta that is bound to _meta and discarded; AppState (lib.rs:71-75) only carries engine, model_name (derived from the file path, not the model), and tokenizer.

Severity: critical.

Missing entirely vs. the goal.md Phase 5 kernel table: q4_gemv, fp16_gemm, softmax (standalone), expert_router. No KernelRegistry, no autotune cache (~/.llm/autotune/*.json), no vectorization-factor/block-size variant selection.
Performance claims — zero backing found anywhere in the repo:

8. "No silent fallback" rule — config.rs and beyond
llm-core/src/model/config.rs::parse_config — re-audited against PROGRESS.md's prior claim that its guessed defaults are "inert":
torch_dtype fallback to WeightDtype::F16 (config.rs:27-33): confirmed still inert for the safetensors/HF loading path — candle.rs:792 unconditionally does meta.weight_dtype = WeightDtype::F16; right after parse_config() returns, overwriting whatever was guessed. PROGRESS.md's prior finding holds.
hidden_act fallback — NOT inert, live and consequential (this was not the field PROGRESS.md's prior review actually checked, despite the general "config.rs reviewed" framing). config.rs:40-43 silently defaults any hidden_act value that doesn't contain the substring "gelu" to HiddenAct::SiLU — including a genuinely missing hidden_act field, or values like "relu"/"geglu"/"swiglu" that are neither gelu-family nor actually SiLU. Unlike weight_dtype, nothing downstream overwrites meta.hidden_act for the safetensors path — it flows straight through to candle.rs:1937 (let act_fn = self.meta.as_ref().map(|m| m.hidden_act).unwrap_or(HiddenAct::SiLU)) and is used to select the actual MLP activation function during the forward pass. Concrete scenario: load an HF safetensors model whose config.json has "hidden_act": "relu" (or omits the field) — the engine will silently run SiLU-gated MLP math instead, producing numerically wrong (not just slightly-off) output with zero warning, because SiLU-gated and ReLU MLPs have different math, not just different nonlinearities. This is exactly the class of bug CLAUDE.md's "no silent fallback" rule and goal.md's "no config files, no model-specific anything" promise are meant to prevent, and it's live on a path (config.json) that goal.md lists as ingestion-priority #1. Compounding: as noted in category 4, this entire HF-safetensors config-parsing path has never been exercised end-to-end against a real checkpoint in this repo's verified history (only GGUF checkpoints were tested) — so this bug (if it is one for some real HF model) would not have been caught by any existing test or manual run.
Severity: high (silent wrong-math fallback, live code path, zero real-world test coverage of the path it lives on).
No other config-parsing code with guessed-default behavior was found outside llm-core (other crates don't parse model config).
Other silent-fallback-adjacent behavior already fixed per PROGRESS.md (unknown GGML quant type now bail!s, missing general.architecture/tokenizer.ggml.tokens now bail!) — spot-checked in candle.rs and confirmed still in place on current master (lines ~405-420, ~611).
